## 1. EXECUTIVE SUMMARY

ROOSTAY is a full-stack rental platform designed specifically for the Ethiopian market. It supports both short-term stays (nightly rates) and long-term rentals (monthly rates), with a payment-first booking flow that integrates Ethiopian payment methods including bank transfer and Telebirr. The platform features a complete admin panel, host dashboard, guest dashboard, review system, and advanced date availability tracking.

### Key Metrics
- **Total Source Files:** ~160
- **API Endpoints:** 50+
- **Database Tables:** 15
- **React Components:** 26
- **CSS Modules:** 8
- **Lines of Code:** ~15,000+

### Core Features
- JWT authentication with httpOnly cookies (XSS-safe)
- Dual booking system (short-term + long-term)
- Payment-first flow with transaction number validation
- Duplicate transaction prevention
- Cloudinary image upload pipeline
- Admin moderation panel
- Host dashboard with earnings tracking
- Advanced date blocking with visual calendar
- Similar listings recommendation engine
- Responsive Airbnb-inspired UI
- Config-driven architecture (zero hardcoded values)

### Data Flow
1. Client makes HTTP request with httpOnly cookies
2. Next.js forwards to Express app via `[...path].js`
3. Express middleware authenticates via cookie or Authorization header
4. Route module validates request with Joi schemas
5. Controller delegates to service layer
6. Service executes parameterized SQL queries
7. Response flows back through the chain

### Key Architectural Decisions
| Decision | Rationale |
|----------|-----------|
| Thin Vercel entry point | 15-line `[...path].js` imports modular Express app |
| httpOnly cookies over localStorage | Eliminates XSS token theft vulnerability |
| Native pg driver over ORM | Full SQL control, no abstraction overhead |
| Modular route structure | Domain-driven design, no merge conflicts |
| Config-driven architecture | All values from environment variables |
| Payment-first booking flow | Prevents unpaid bookings from consuming availability |
| Serverless expiry via on-read cleanup | No cron jobs needed in serverless environment |

### 6.1 Configuration System

All configuration is stored in JSON files inside `packages/config/` and loaded via `packages/config/index.js`. The loader resolves environment-specific sections (development/production/test) and replaces UPPER_CASE string values with matching `process.env` variables.

**Config Resolution Pattern:**
```javascript
// config value in JSON:
{ "host": "DB_HOST" }

// Resolved at runtime:
process.env.DB_HOST // e.g., "ep-steep-hall-as1pk0kp-pooler.eu-central-1.aws.neon.tech"
```

**Key Environment Variables:**
| Variable | Required | Purpose |
|----------|----------|---------|
| `DATABASE_URL` | Yes | PostgreSQL connection string (Neon) |
| `JWT_SECRET` | Yes | JWT access token signing secret |
| `JWT_REFRESH_SECRET` | Yes | JWT refresh token signing secret |
| `JWT_EXPIRES_IN` | No | Access token expiry (default: 30m) |
| `JWT_REFRESH_EXPIRES_IN` | No | Refresh token expiry (default: 7d) |
| `CLOUDINARY_CLOUD_NAME` | No | Cloudinary cloud name for image uploads |
| `CLOUDINARY_API_KEY` | No | Cloudinary API key |

### 6.3 Authentication & Security

**Token Flow:**
1. User registers/logs in → server generates access token (30min) + refresh token (7 days)
2. Tokens set as httpOnly cookies (`roostay_access_token`, `roostay_refresh_token`)
3. Browser automatically sends cookies on every request
4. `authenticate` middleware reads access token from cookie (priority 1) or Authorization header (priority 2)
5. If access token expired → frontend calls `/api/auth/refresh-token` with refresh cookie
6. Server issues new token pair, sets new cookies
7. Logout clears both cookies

**Security Measures:**
- httpOnly cookies prevent XSS token theft
- SameSite=Lax prevents CSRF
- Secure flag in production (HTTPS only)
- Parameterized SQL queries prevent SQL injection
- Joi validation on all inputs
- Helmet security headers
- Rate limiting with Redis/in-memory fallback
- bcrypt password hashing (12 rounds)
- Account lockout after 5 failed login attempts

### 6.4 Middleware Pipeline

**Order of Execution:**
1. `helmet` — Security headers
2. `cors` — Cross-origin resource sharing
3. `express.json` — JSON body parsing (10MB limit)
4. `express.urlencoded` — URL-encoded parsing
5. `cookieParser` — Cookie parsing
6. Route-specific middleware:
   - `rateLimiter` — Rate limiting per route type
   - `authenticate` — JWT verification
   - `authorize` — Role-based access
   - `validateRequest` — Joi schema validation
7. Route handler (controller)
8. `notFoundHandler` — 404 catch-all
9. `errorHandler` — Global error handler

### 7.1 Page Architecture

The frontend utilizes Next.js 14's **App Router** for page rendering, leveraging React Server Components (RSC) where possible, and Client Components (`'use client'`) for interactive UI.

- **Route Groups**: `(auth)` is used to isolate authentication pages with a minimal layout (no header/footer).
- **Role-Based Routing**: Strict path segregation ensures clear mental models:
  - `/guest/*`: Booking history, favorites, profile management.
  - `/host/*`: Dashboard, listing CRUD, booking management, withdrawals.
  - `/admin/*`: Platform moderation, user management, financial oversight.
- **API Catch-All**: `frontend/pages/api/[...path].js` acts as the **sole Vercel serverless entry point**. It is strictly a 15-line thin wrapper that imports and mounts the modular Express application from `packages/api/index.js`. This prevents code duplication and ensures the monorepo's backend logic is the single source of truth.

### 7.2 Component Library

Components are strictly categorized by responsibility to maintain scalability:

1. **Layout Components** (`components/layout/`): `Header.jsx`, `Footer.jsx`, `Sidebar.jsx`, `Breadcrumb.jsx`. Handle global navigation, responsive hamburger menus, and role-adaptive link rendering.
2. **UI Primitives** (`components/ui/`): Atomic, reusable elements (`Button`, `Input`, `Modal`, `Card`, `Badge`, `Skeleton`, `Avatar`, `StarRating`, `Pagination`, `Toast`, `CountdownTimer`). Designed with prop-driven variants (e.g., `variant="primary" | "outline" | "danger"`).
3. **Feature Components** (`components/listing/`, `components/booking/`, etc.): Molecular components composing UI primitives and business logic. 
   - *Critical Fix Implemented*: `ImageGallery.jsx` and `ListingCard.jsx` now include robust fallback logic (`img.url || img.imageUrl || img.image_url || PLACEHOLDER`) to seamlessly handle both snake_case (database direct) and camelCase (frontend transformed) image payloads, preventing broken image icons.
   - `BookingCard.jsx` includes guard clauses for undefined `listing` props and strictly enforces the `isOwnListing` check to prevent hosts from booking their own properties.

### 7.3 Custom Hooks

- `useAuth.js`: Manages authentication state. Fetches `/api/auth/me` on mount. Exposes `user`, `isAuthenticated`, `isGuest`, `isHost`, `isAdmin`, `login`, `register`, and `logout`.
- `useApi.js`: A generic, promise-based API wrapper handling loading states, error catching, and data return. Integrates with the centralized `apiClient` for automatic token refresh queuing.
- `useBookings.js` / `useListings.js` / `useFavorites.js`: Domain-specific data fetching hooks that wrap `useApi` to provide typed, predictable data shapes to their respective pages.

### 7.4 State Management

ROOSTAY avoids heavy client-side stores (like Redux) in favor of:
1. **React Context**: Used sparingly for global UI states (`ToastContext` for notifications, `ConfirmContext` for destructive action modals).
2. **Component State**: `useState` and `useReducer` for local form inputs and modal visibility.
3. **Server State**: Managed via the `useApi` hook pattern, ensuring data is fetched fresh or cached predictably without complex cache invalidation logic.

### 7.5 CSS Design System

The styling is built on **Pure CSS** (no Tailwind or Bootstrap) to ensure zero bundle bloat and maximum customization. It follows a modular, BEM-inspired architecture:
- `variables.css`: CSS custom properties for colors (`--color-primary`, `--color-warning`), typography (`--font-size-xl`), spacing, and shadows.
- `reset.css`: Modern CSS reset normalizing browser defaults.
- `utilities.css`: Helper classes for flexbox, grid, text alignment, and spacing.
- `components.css`: Styles for all UI primitives.
- `layout.css` & `pages.css`: Structural and page-specific styling.
- `responsive.css`: Mobile-first media queries ensuring flawless rendering from 320px mobile screens to 4K desktops.

### 7.6 Responsive Design

- **Mobile-First**: Base styles target mobile devices; `@media (min-width: 768px)` and `@media (min-width: 1024px)` handle tablet and desktop layouts.
- **Adaptive Navigation**: The `Header` component collapses into a hamburger menu on mobile, while displaying full horizontal navigation and user dropdowns on desktop.
- **Grid/Flex Fallbacks**: Listing grids automatically adjust from 1 column (mobile) to 2 (tablet) to 3-4 (desktop) using CSS Grid `repeat(auto-fill, minmax(300px, 1fr))`.

## 9. DATABASE DOCUMENTATION

- **Driver**: Native `pg` (node-postgres) for maximum performance and zero ORM abstraction overhead.
- **Connection Pooling**: Singleton `Pool` instance configured for Neon serverless compatibility (`ssl: { rejectUnauthorized: false }`, `idleTimeoutMillis: 30000`).
- **Security**: **100% parameterized queries** (`$1, $2`) to eliminate SQL injection risks.
- **Data Integrity**: 
  - `safeTruncate(str, maxLength)` utility applied to all `VARCHAR` inserts (e.g., `city` max 100, `street_address` max 500) to prevent "value too long" database crashes.
  - Foreign key constraints with `ON DELETE CASCADE` where appropriate (e.g., `listing_images` deleting when parent `listing` is deleted).
- **Serverless Expiry**: The `bookings` table includes a `payment_expires_at` timestamp. The `expireUnpaidBookings()` function runs on every booking fetch to clean up stale pending bookings without requiring external cron jobs.

## 10. PAYMENT FLOW ARCHITECTURE

ROOSTAY utilizes a **Payment-First, Manual Verification** flow tailored for the Ethiopian market (Telebirr, Commercial Bank of Ethiopia):

1. **Selection**: Guest selects dates/guests on `ListingDetailPage`. `BookingCard` calculates estimated total (Base + Cleaning + 5% Service Fee).
2. **Initiation**: Guest clicks "Book Now". `BookingPaymentModal` opens, displaying official bank/Telebirr details and a 30-minute countdown timer.
3. **External Payment**: Guest switches apps, makes the transfer, and copies the Transaction Reference Number.
4. **Validation**: Guest enters the reference number in the modal. Frontend calls `/api/payments/validate-transaction` to ensure it hasn't been used (preventing duplicate bookings).
5. **Submission**: Frontend calls `POST /api/bookings`. Backend creates:
   - A `booking` record with `status = 'pending'` and `payment_expires_at = NOW() + 30 mins`.
   - A `payment` record with `status = 'processing'` and `transaction_reference = <input>`.
6. **Verification**: Admin views the pending payment in `/admin/payments`, matches the transaction ID with their bank statement, and clicks "Verify".
7. **Confirmation**: Backend updates payment to `completed` and automatically cascades the booking status to `confirmed`. Guest and Host receive notifications.

## 13. KNOWN ISSUES & RESOLUTIONS (Migration Log)

During the transition from a 2,000-line monolith to a modular architecture, the following critical bugs were identified and permanently resolved:

1. **Route Collision (500 Internal Server Error)**: 
   - *Issue*: `GET /api/listings/host` was returning a PostgreSQL UUID parsing error.
   - *Root Cause*: Express matches routes sequentially. `app.get('/api/listings/:id')` was registered *before* `/host`, so "host" was treated as the `:id` parameter.
   - *Resolution*: Moved static routes (`/host`, `/search`) **above** parameterized routes (`/:id`) in `packages/api/routes/index.js`.
2. **Image Rendering Failures**:
   - *Issue*: Listing cards and galleries showed broken image icons.
   - *Root Cause*: Database returns `image_url` (snake_case), but frontend components expected `imageUrl` (camelCase).
   - *Resolution*: Implemented bulletproof fallback mapping in `ImageGallery.jsx` and `ListingCard.jsx`: `const url = img.url || img.imageUrl || img.image_url || PLACEHOLDER;`.
3. **Duplicate Hook Execution**:
   - *Issue*: React hydration warnings in `BookingCard.jsx`.
   - *Root Cause*: `useAuth()` was accidentally called twice in the same component scope.
   - *Resolution*: Consolidated to a single `const { isAuthenticated, user } = useAuth();` call.
4. **Monolith Code Duplication**:
   - *Issue*: Backend logic existed in both `packages/api/` and `frontend/pages/api/[...path].js`.
   - *Resolution*: Refactored `[...path].js` into a 15-line thin entry point that solely imports and executes `createApp()` from `packages/api/index.js`.
